# spec-revise — Reviser

**Role**: reviser (one reviser consumes N critic siblings — here the review + audit pair, the `report`/`primer` shape).
**Reads**: latest `<thread>.{N}/<thread>.tex` (+ `sections/*.tex`) + `_progress.json`, BOTH `<thread>.{N}.review/` and `<thread>.{N}.audit/` (all files), the resolved `code_ref` implementation (when active), `<thread>/refs/` + shared `research/`, project `BRIEF.md`.
**Writes**: `<thread>.{N+1}/` with `<thread>.tex` (+ `sections/*.tex`), `changelog.md`, `_progress.json` — or reports `AUDITED` without writing when the combined verdict pre-check passes.

## CLI flags

### `--override-code-wrong "<reason>"` (optional — the code-wrong escalation override)

The friction-ful override for a `spec-audit` `implementation_contradicts_spec` finding whose `Disposition` is **`code-wrong`** (SKILL.md §Audit verdict, `spec-audit.md` §The three-way verdict). A `code-wrong` finding means the spec is the source of truth (a ratified ADR) and the *implementation* has drifted — so `spec-revise` **MUST NOT** "fix" it by rewriting the spec to match the code. By default the finding **blocks advance** (step 2): the reviser refuses to terminal-exit and refuses to reconcile the finding toward the code. The operator has exactly two ways forward:

1. **Fix the code** (out of band, in the consumer repo) and re-run `spec-audit` — once the code matches the spec, the finding no longer fires and the thread advances normally. This is the intended path.
2. **Override with `spec-revise <thread> --override-code-wrong "<reason>"`** — the operator explicitly disagrees with the escalation (e.g. they have decided the spec claim WILL be relaxed to match the code, converting a `code-wrong` into a deliberate `spec-wrong`). This is modeled on `memo`'s `--override-no-go "<reason>"` (SKILL.md §NO-GO terminal state): **the reason argument is required** — absent / empty (`--override-code-wrong ""`) / whitespace-only reasons are rejected and the thread is left untouched (no version dir, no `_progress.json` mutation). The override does NOT silently drop the finding's severity; it records the operator's explicit, durable rationale.

- **What the override permits:** the reviser MAY, on this pass only and ONLY for the overridden finding, reconcile the spec claim toward the code (the operator has taken responsibility for the direction). Every other critical flag (from either critic) still binds. The escalation block in the audit `verdict.md` is preserved as a permanent record of the code-wrong recommendation.
- **Audit trail** (step 9): `metadata.code_wrong_overridden = true` + `metadata.code_wrong_override_reason = "<verbatim reason>"`. Durable — carried on every subsequent shallow merge, exactly as `memo`'s `metadata.no_go_overridden` / `metadata.no_go_override_reason`.
- **Never silently rewrites.** Without the override, a `code-wrong` finding is an operator decision the reviser records as `escalated — operator decision required` in `changelog.md` and does NOT reconcile toward the code (`datasheet-revise.md` "fix to the source, not the critic" discipline; the near-miss the class exists to prevent).

### `--polish "<reason>"` (optional)

Operator-directed revision entry point — the sanctioned, audit-trailed path for spending one additional revision pass when the combined verdict pre-check (step 2) would otherwise force a terminal exit. Full contract: `anvil/lib/snippets/directed_revision.md` (`.anvil/anvil/lib/snippets/directed_revision.md` in an installed consumer repo). Summary:

- **Bypasses step 2 ONLY.** When passed, the combined verdict pre-check is skipped, so the reviser runs against an `AUDITED`-terminal version (which the default path correctly refuses) and polishes sub-threshold per-dimension justifications in `scoring.md`, `nit`-tagged or untagged `comments.md` notes, and audit-side line-level findings.
- **The critic-completeness check (step 1) still applies.** `--polish` bypasses the pre-check *verdict*, never the *existence* of the critics — BOTH a completed `<thread>.{N}.review/` AND a completed `<thread>.{N}.audit/` are still required.
- **The iteration-cap check (step 3) still applies.** `--polish` against a thread at `max_iterations` still hits the `BLOCKED` notice.
- **The reason argument is required.** `--polish` with no value, `--polish ""`, and `--polish "   "` (whitespace-only) are all rejected with a clear error; the thread is left untouched (no version dir written, no `_progress.json` mutation).
- **No inherited credit.** The polish-pass output is a normal `<thread>.{N+1}/` version dir. The next `spec-review` + `spec-audit` pass scores it on its own rubric merits — a fresh critic pair MUST land for the thread to re-reach `AUDITED`. The critics do NOT read the audit-trail fields and do NOT special-case the polish pass.
- **Audit-trail fields** (step 9): `metadata.revision_mode = "polish"` + `metadata.revise_force_reason = "<verbatim reason>"`, both audit-trail-only (NOT scored, NOT gating, NO state-machine impact).

See SKILL.md §"Operator-initiated polish passes" for the user-facing shape.

## Procedure

1. **Discover state**: find the highest `N` with `<thread>.{N}/<thread>.tex`. Require BOTH a completed `<thread>.{N}.review/` AND a completed `<thread>.{N}.audit/` (else exit pointing at the missing critic — `REVIEWED-PARTIAL`/`AUDITED-PARTIAL` are not advance-eligible per SKILL.md). Require `<thread>.{N+1}/` to not exist (immutability — never revise in place).
2. **Combined verdict pre-check**: read `<thread>.{N}.review/verdict.md` and `<thread>.{N}.audit/verdict.md`. When the review records `advance: true` (total ≥39/44, zero unresolved review critical flags) AND the audit records `audit_clean: true` (zero unresolved `implementation_contradicts_spec` critical flags), the thread is **`AUDITED` — terminal**: report the publish-handoff summary (resolved body path, review total /44, clean audit, the handoff guarantees per SKILL.md §Publish handoff contract; note that `spec-figures` may optionally produce the PDF) and exit WITHOUT writing a new version.

   **`code-wrong` block**: an audit `implementation_contradicts_spec` finding with `Disposition: code-wrong` sets `audit_clean: false` and **blocks terminal exit AND blocks reconciliation toward the code**. The reviser does NOT rewrite the spec to match the code (that is the near-miss). It exits reporting the escalation UNLESS the operator passed `--override-code-wrong "<reason>"` (see §CLI flags) — which permits reconciling that one finding toward the code on this pass only. An `unregistered` intentional-gap finding also sets `audit_clean: false`, but its fix is drafter/operator-side (add the register row), not a code reconciliation — see step 6.

   **`--polish` bypass** (`anvil/lib/snippets/directed_revision.md`): when `spec-revise <thread> --polish "<reason>"` is invoked, this step is skipped entirely — proceed to step 4 regardless of `advance: true` + clean audit. Pre-check the reason argument before bypassing: an absent / empty / whitespace-only reason is rejected with a clear error and the thread is left untouched. `--polish` bypasses ONLY this step — step 1's dual-critic-required check and step 3's iteration cap still apply. **`--polish` does NOT bypass a `code-wrong` block** — a `code-wrong` escalation still requires `--override-code-wrong` (they are independent overrides). See §"CLI flags" for the full required-reason + no-inherited-credit contract.
3. **Iteration-cap check**: default `max_iterations: 4` (worst-case terminal version `<thread>.5/`); project-BRIEF paired override (`max_iterations` + `iteration_cap_rationale`) per the #349 memo contract — the BLOCKED notice surfaces the rationale verbatim when an elevated cap is hit. At cap → report `BLOCKED — human review required` and exit.
4. **Read all critic input**: from the review — `verdict.md` (top revision priorities first), `scoring.md` (per-dim deductions; dim 1 normative-correctness gaps lead), `comments.md` (severity + `scope` tags), and the "What's working" list. From the audit — `verdict.md` (critical flags first, then `major` findings), `findings.md` (per-claim factual + implementation-consistency findings), `comments.md`. The two verdicts combine: a critical flag from *either* critic blocks.
5. **Handle the code_ref contract (conditional)**: when the BRIEF declares an active `code_ref`, re-resolve it (`anvil/lib/project_brief.py::resolve_code_ref(<project_dir>, <slug>)`) and read the implementation alongside the critic feedback so the revision stays consistent with it. When either critic carried the missing/unresolvable-`code_ref` `major` finding, surface it in the report (the fix is operator-side BRIEF authoring or path correction, not body editing).
6. **Build the revision plan**, ordered: (1) critical flags — every flag from EITHER critic MUST be addressed:
   - **Self-contradiction** (review-side) → reconcile the two incompatible statements to one normative value (choosing the correct one against `code_ref` when active, or escalating to the operator when the correct value is itself ambiguous).
   - **Undefined normative term** (review-side) → add the missing definition (in a definitions/notation section), not by deleting the normative use.
   - **`implementation_contradicts_spec`** (audit-side critical flag) — **route by the finding's `Disposition`; NEVER apply a blanket "match the code" fix**:
     - **`spec-wrong`** → the code is the ratified truth and the spec claim is stale. Fix the spec claim to match the resolved `code_ref` (the normal revise path). Quote the corrected value against the code in `changelog.md`.
     - **`code-wrong`** → the spec is the source of truth and the code has drifted. **Do NOT rewrite the spec toward the code.** Record it in `changelog.md` as `escalated — operator decision required` and surface the audit's escalation block in the report; the finding blocks advance until the code is fixed (re-run `spec-audit`) or the operator passes `--override-code-wrong "<reason>"` (§CLI flags). This is the load-bearing safety property — the near-miss the class exists to prevent (`datasheet-revise.md` "fix to the source, not the critic").
     - **`intentional-gap` + `unregistered`** → the divergence is an accepted target-vs-live gap that was never registered. The fix is **NOT a spec-claim edit and NOT a code reconciliation** — add (or ask the operator/drafter to add) an `## Implementation status` register row for the affected component (component / live / target / status: target-state / tracking). Once the row exists, the next `spec-audit` suppresses the contradiction. Record the register addition in `changelog.md`.
     - **Uncertain disposition?** The auditor already applied the asymmetry default (`code-wrong` under uncertainty); the reviser honors that — it does not silently re-classify a `code-wrong` finding as `spec-wrong` to make it go away.
   (2) `blocker`/`major` comments (including review's step-5b unregistered-target-state `major` finding → add the register row); (3) the lowest-scoring dims' deductions; (4) `minor`/`nit` only when they don't conflict with (1)–(3). Never touch the "What's working" list — the normative statements the reviewer flagged as load-bearing.
7. **Write `<thread>.{N+1}/<thread>.tex`** (+ `sections/*.tex`, slug-echo per #295) applying the plan. Re-run the drafter's self-disciplines on the result (internal-consistency sweep, code_ref-correspondence check) — the revision must not introduce a fresh instance of the failure mode it just fixed.
   - **Preserve/update the figure plan (draft-time figure-reference contract)**: carry forward the drafter's figure references and the `metadata.figure_plan` record. When the revision reorders sections or a critic flagged a figure's caption/placement, update the references AND the plan together so the two stay in sync. Adding a newly-needed diagram means placing a new reference + plan entry as the drafter's step 5 prescribes; removing a section that owned a figure removes both. `spec-figures` re-renders to whatever paths the revised body now references. Zero-figure threads carry an empty/absent plan forward unchanged (silent-off).
8. **Write `changelog.md`** mapping each consumed critic note to the change made (or to an explicit `declined — <reason>` entry; scoring deductions may be argued against, critical flags may not). **A `code-wrong` `implementation_contradicts_spec` finding the reviser did NOT reconcile is recorded as an explicit `escalated — operator decision required` entry** (with the audit's suggested consumer-repo issue text), never as a silent spec rewrite; a `spec-wrong` finding is recorded as `fixed spec claim to match code_ref <impl:line>`; an `unregistered` intentional-gap is recorded as `added ## Implementation status register row for <component>`. On a `--override-code-wrong` pass, record `code-wrong reconciled per operator override — reason: "<verbatim>"`. On a `--polish` pass, prepend a blockquote header note quoting the operator's `--polish` reason verbatim, and map each polish edit to its source (a sub-threshold dimension deduction, a `nit`/untagged comment, or an audit finding) or to the operator directive. The prior review's "What's working" list still binds.
9. **Initialize `_progress.json`** for the new version: `phases.revise.state = done` (LAST write), carry forward `metadata.code_ref_resolved` (when active) and `metadata.figure_plan` (updated per step 7), and **append the `score_history` row** for the completed review iteration per `anvil/lib/snippets/progress.md` §Convergence fields: `{ "iteration": <N>, "total": <reviewed-total>, "threshold": 39, "rubric_id": "anvil-spec-v1" }`. Stable-score termination (`STALLED`) follows `anvil/lib/snippets/rubric.md` §"Termination resolution order" over this history.

   **`--override-code-wrong` audit trail**: on a code-wrong-override pass, additionally write `metadata.code_wrong_overridden = true` and `metadata.code_wrong_override_reason = "<verbatim operator-supplied reason>"` (stored verbatim, carried on every subsequent shallow merge — the durable record that the operator took responsibility for reconciling a `code-wrong` finding toward the code). On the default path these are omitted / `false` / `null`.

   **Polish-pass audit trail** (`anvil/lib/snippets/directed_revision.md` §"Audit-trail fields"): on a `--polish` pass, additionally write `metadata.revision_mode = "polish"` and `metadata.revise_force_reason = "<verbatim operator-supplied reason>"` (stored verbatim). Both fields are audit-trail-only: NOT scored, NOT gating, NO state-machine impact. On the default (no-`--polish`) path, `revision_mode` defaults to `"normal"` (or is omitted) and `revise_force_reason` is `null` (or omitted).
10. **Report**: e.g., `Revised botho-consensus.1 → botho-consensus.2 (fixed 2 spec-wrong claims to match code_ref, added 1 register row for an unregistered gap; 1 code-wrong finding escalated to operator, not rewritten). Next: spec-review + spec-audit botho-consensus`.

## What spec-revise does NOT do

- **Never edits `<thread>.{N}/` or any critic sibling in place** — immutability is the audit trail.
- **Never rewrites the spec to match a vestigial code path.** A `code-wrong` `implementation_contradicts_spec` finding is escalated to the operator and blocks advance; it is reconciled toward the code ONLY under an explicit `--override-code-wrong "<reason>"`. This is the load-bearing safety property of the class — the near-miss it exists to prevent.
- **Never re-classifies a critic's disposition to make a finding go away.** The auditor's `Disposition` (spec-wrong / code-wrong / intentional-gap) is honored; the reviser does not downgrade a `code-wrong` to `spec-wrong` to avoid the escalation.
- **Never populates the register on the auditor's behalf beyond the drafter role** — adding an `## Implementation status` row to resolve an `unregistered` gap is a legitimate reviser edit (it is body content), but the reviser never invents a target-state gap that the critics did not identify.
- **Never advances state itself** — the next `spec-review` + `spec-audit` pass scores `<thread>.{N+1}/` on its own merits.
- **Never bypasses critical flags** — a changelog `declined` entry is legitimate for scoring deductions, never for a critical flag from either critic.

## Git sync (opt-in, off by default)

Per `anvil/lib/snippets/git_sync.md`: if `.anvil/config.json` exists and `git.commit_per_phase` is `true`, end this phase: stage only the dirs this phase wrote, commit as `anvil(<skill>/<phase>): <thread>.{N} [<state>]`, push if `git.push` is `true`. Git failures warn and continue. Default off.

This phase's specifics:

- **Ordering**: after the `_progress.json` `done` write lands. On the no-write paths (AUDITED / BLOCKED at step 2–3) there is nothing to commit and the hook is a silent no-op.
- **Staging target**: ONLY this command's own `<thread>.{N+1}/` version dir.
- **Commit**: `anvil(spec/revise): <thread>.{N+1} [REVISED]`.
